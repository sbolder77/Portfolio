Discussion 1 - Part 1 - Initial Post

The OWASP coding weakness I have focussed on in my discussion is A09:2021-Security Logging and Monitoring Failures. I have represented the flow of a sample registration using an activity diagram as a high-level process flow. The activity diagram highlights the steps that would be executed in a laboratory for logging and testing a sample as well as sending results. By only logging the select events of authentication and logging in a sample it fails to identify all the logging or monitoring that would be required through the process which would be recognised as a weakness and broken. With the sensitivity and criticality of the process and data, it should be assumed that security logging and monitoring would take place across all steps. By being able to complete the sample after authentication the software bypasses authorisation and emailing sample data is also not logged so the software misses crucial auditing and system logs/monitoring.

In addition to an activity diagram use case diagrams would be an appropriate choice to successfully represent the actors required for consuming and actioning logging and monitoring events that are captured through the software. Furthermore, a sequence diagram would support the messages and their states through the software transaction process. The activity diagram could maintain the process flow and the supporting diagrams would surface the objects and states to log and monitor.
